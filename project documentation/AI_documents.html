Views

The application's views handle the API logic. They are defined in views.py.

DocumentUploadView

This view handles the uploading and processing of documents.

Methods:

  • post(self, request, *args, **kwargs): Handles the POST request for uploading a document. It performs the following steps:
    1. Receives the uploaded file.
    2. Saves the file temporarily.
    3. Uses DocumentProcessor to extract text from the document.
    4. Cleans the extracted text.
    5. Uses convert_text_to_json to convert the cleaned text to structured JSON.
    6. Saves the structured data into the Applicant model.
    7. Deletes the temporary file.
    8. Returns a response with the processing results.

ApplicantListView

This view lists all the applicants.

Methods:

  • get(self, request, *args, **kwargs): Handles the GET request to retrieve a list of all applicants. It returns a summary of each applicant's data.

ApplicantDetailView

This view retrieves the detailed information of a specific applicant.

Methods:

  • get(self, request, applicant_id, *args, **kwargs): Handles the GET request to retrieve the full details of a single applicant, identified by applicant_id.

ConvertApplicantToUserView

This view converts an existing Applicant instance into a Users instance.

Methods:

  • post(self, request, *args, **kwargs): Handles the POST request to convert an applicant to a user. It takes an applicant_id as input.

BatchConvertApplicantsView

This view converts multiple applicants to users in a batch operation.

Methods:

  • post(self, request, *args, **kwargs): Handles the POST request for batch conversion. It accepts a list of applicant_ids.

SyncStatusView

This view checks the synchronization status between the Applicant and Users models.

Methods:

  • get(self, request, *args, **kwargs): Handles the GET request to check the sync status. It takes an applicant_id as a query parameter.

Document Processing

The application includes a robust document processing pipeline to handle PDF and DOCX files. This functionality is primarily managed by the DocumentProcessor class in document_processor.py.

DocumentProcessor

The DocumentProcessor class is responsible for extracting text and metadata from documents.

Key Features:

  • Support for Multiple Formats: It can process both PDF and DOCX files.
  • Resilient PDF Processing: For PDFs, it first tries to use PyMuPDF (fitz) for its robustness. If that fails, it falls back to PyPDF2, and as a last resort, it uses pdfplumber.
  • Comprehensive DOCX Processing: For DOCX files, it extracts text from both paragraphs and tables.
  • Metadata Extraction: It extracts metadata such as page count and word count.

Methods:

  • process_document(self, file_path, document_type=None): The main method that takes a file path and an optional document type, and returns a dictionary with the extracted text and metadata.
  • _detect_document_type(self, file_path): Detects the document type based on the file extension.
  • _process_pdf(self, file_path): The internal method for processing PDF files.
  • _process_docx(self, file_path): The internal method for processing DOCX files.
  • get_document_info(self, file_path): A utility method to get basic information about a document without full processing.

AsyncDocumentProcessor

To handle large documents without blocking the main thread, the application provides an AsyncDocumentProcessor class. This class wraps the synchronous DocumentProcessor and runs its methods in a thread pool, making the processing non-blocking.

JSON Conversion

The document_to_json.py module is responsible for converting the raw text extracted from documents into a structured JSON format. This is a crucial step in the data processing pipeline, as it transforms unstructured text into a machine-readable format that can be saved in the database.

convert_text_to_json

This is the primary function that orchestrates the JSON conversion process.

Key Features:

  • LLM-Powered Extraction: It uses a Large Language Model (LLM) to understand the context of the extracted text and map it to a predefined JSON schema.
  • Robust JSON Repair: It includes a repair_json_string function to fix common JSON formatting errors that can occur in the LLM's output.
  • Manual Extraction Fallback: In cases where the JSON output is severely malformed and cannot be repaired, it falls back to a extract_json_data_manually function that uses regular expressions to extract key information.

Process:

  1. The function takes the cleaned, extracted text as input.
  2. It sends the text to the LLM with a prompt that specifies the desired JSON structure.
  3. The LLM's raw output is then cleaned and parsed.
  4. If parsing fails, it attempts to repair the JSON string.
  5. If repair fails, it uses manual extraction as a fallback.
  6. The final, structured JSON data is returned as a Python dictionary.

Data Mapper Service

The data_mapper_service.py module provides the logic for converting the JSON data extracted from documents into the format required by the Users model in the api app. This service is essential for integrating the document processing functionality with the main user management system.

DataMapperService

This class contains a collection of static methods that handle the data mapping and transformation.

Key Responsibilities:

  • Date Parsing: The parse_date_string method can parse various date formats commonly found in documents, ensuring that date information is correctly stored.
  • Certificate Extraction: The extract_certificates_from_data method intelligently extracts certificate names from multiple sections of the structured data.
  • Reference and Sea Service Extraction: The extract_references_from_data and extract_sea_services_from_data methods extract and format the applicant's references and sea service history.
  • Applicant to User Mapping: The core logic resides in the map_applicant_to_users method, which takes an Applicant instance and maps its data to a new Users instance. This method handles the complex task of flattening the JSON data and assigning it to the appropriate fields in the Users model.
  • Batch Processing: The convert_and_save_applicant and batch_convert_applicants methods provide the functionality to convert and save single or multiple applicants to the Users model, ensuring data consistency through the use of database transactions.

Architecture Overview

This Django REST Framework application is designed to process documents, extract structured data, and integrate that data with a user management system. The architecture follows a modular design, with clear separation of concerns across different components.

High-Level Architecture

The application operates through a multi-stage pipeline that transforms uploaded documents into structured database records. When a user uploads a document, the system first validates the file format and size through the serializer layer. The validated file is then passed to the document processor, which extracts raw text from PDF or DOCX files using multiple fallback strategies to ensure robustness.

Once the text is extracted, the system employs a cleaning mechanism to remove duplicate lines, boilerplate content, and other noise that might have been introduced during the extraction process. The cleaned text is then fed into a Large Language Model that has been specifically prompted to extract structured information according to a predefined schema. This schema maps to the various sections of a typical seafarer application form, including personal details, education, professional qualifications, and sea service history.

The structured data is stored in the Applicant model using JSON fields, which provides flexibility for handling varying data structures. The application also includes a data mapping service that can transform the JSON-based applicant data into a more rigid, field-based structure in the Users model of a separate API application. This dual-storage approach allows the system to maintain both the raw extracted data and a normalized representation suitable for integration with other systems.

Component Interaction

The views serve as the entry points for all API operations. The DocumentUploadView orchestrates the entire document processing pipeline, coordinating between the document processor, the JSON converter, and the database models. The applicant management views provide CRUD operations for accessing and managing the extracted data. The conversion views handle the transformation of applicant data into user records, utilizing the data mapper service to ensure accurate field mapping.

The serializers act as a validation and transformation layer between the HTTP requests and the internal business logic. They ensure that incoming data meets the required format and constraints before it reaches the views. The serializers also format the outgoing responses, providing a consistent API interface to clients.

The document processor is designed with resilience in mind, implementing multiple fallback strategies for PDF processing. This ensures that the system can handle a wide variety of document formats and structures, even when individual processing libraries fail. The asynchronous wrapper allows the system to handle large documents without blocking the main request thread, improving overall system responsiveness.

Data Flow

The typical data flow for a document upload operation follows this sequence:

  1. A client sends a POST request to the /upload/ endpoint with a document file.
  2. The DocumentUploadSerializer validates the file format and size.
  3. The DocumentUploadView saves the file temporarily and invokes the DocumentProcessor.
  4. The DocumentProcessor extracts raw text from the document using the appropriate library.
  5. The extracted text is cleaned to remove duplicates and boilerplate content.
  6. The cleaned text is passed to the convert_text_to_json function, which uses an LLM to structure the data.
  7. The structured JSON is validated and saved to the Applicant model.
  8. The temporary file is deleted, and a response is returned to the client with the processing results.

For conversion operations, the data flow is:

  1. A client sends a POST request to the /convert/ endpoint with an applicant ID.
  2. The ConvertApplicantRequestSerializer validates the request.
  3. The ConvertApplicantToUserView retrieves the Applicant instance from the database.
  4. The DataMapperService maps the applicant's JSON data to the Users model fields.
  5. A new Users instance is created and saved to the database.
  6. A success response is returned to the client.

Design Patterns

The application employs several design patterns to maintain code quality and extensibility. The Service Layer pattern is evident in the DataMapperService and the document processing logic, which encapsulates complex business logic outside of the views. The Strategy Pattern is implemented in the PDF processing, where multiple extraction strategies are tried in sequence until one succeeds. The Template Method pattern is used in the document processor, where the main processing flow is defined but specific steps are delegated to format-specific methods.

Installation and Setup

This section provides instructions for setting up the application in a development environment.

Prerequisites

Before installing the application, ensure that you have the following prerequisites:

  • Python 3.8 or higher
  • Django 4.0 or higher
  • Django REST Framework 3.14 or higher
  • PostgreSQL or another database supported by Django

Installing Dependencies

The application requires several Python packages for document processing and LLM integration. Install these dependencies using pip:

pip install PyPDF2 PyMuPDF python-docx pdfplumber langchain langchain-ollama

Database Configuration

Configure your database settings in the Django settings.py file. The application uses standard Django models, so any database supported by Django will work. For production environments, PostgreSQL is recommended.

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'your_database_name',
        'USER': 'your_database_user',
        'PASSWORD': 'your_database_password',
        'HOST': 'localhost',
        'PORT': '5432',
    }
}

Running Migrations

After configuring the database, run the Django migrations to create the necessary tables:

python manage.py makemigrations ai_document
python manage.py migrate

LLM Configuration

The application uses Ollama for LLM-powered JSON extraction. Ensure that you have Ollama installed and running on your system. The default model used is llama3.2:1b, but this can be changed in the document_to_json.py file.

To install Ollama, follow the instructions at the official Ollama website.

Once installed, pull the required model:

ollama pull llama3.2:1b

Media Files Configuration

The application handles file uploads, so you need to configure the media files settings in settings.py:

MEDIA_URL = '/media/'
MEDIA_ROOT = os.path.join(BASE_DIR, 'media')

Ensure that the media directory exists and is writable by the Django application.

URL Configuration

Include the application's URLs in your main Django project's urls.py file:

from django.urls import path, include

urlpatterns = [

Running the Development Server

Start the Django development server:

python manage.py runserver

The application will be accessible at http://localhost:8000/ai_document/.

Error Handling and Troubleshooting

The application includes comprehensive error handling to manage various failure scenarios gracefully.

Common Errors and Solutions

File Upload Errors

If you encounter errors during file upload, check the following:

  • File Size: Ensure that the file size does not exceed 50MB, as this is the maximum allowed size.
  • File Format: Only PDF and DOCX files are supported. Other formats will be rejected.
  • File Permissions: Ensure that the Django application has write permissions to the media directory.

Document Processing Errors

Document processing can fail for various reasons:

  • Corrupted Files: If a file is corrupted or malformed, the document processor may fail to extract text. The application will log the error and return a detailed error message.
  • Missing Dependencies: If the required PDF or DOCX processing libraries are not installed, the processor will raise a DocumentProcessingError. Ensure that all dependencies are installed correctly.
  • Unsupported Document Structure: Some documents may have complex structures that are difficult to parse. In such cases, the extracted text may be incomplete or contain errors.

JSON Conversion Errors

The JSON conversion process relies on an LLM, which can sometimes produce malformed output:

  • LLM Unavailable: If the Ollama service is not running, the conversion will fail. Ensure that Ollama is installed and running on your system.
  • Malformed JSON: The application includes robust JSON repair mechanisms, but in extreme cases, the output may still be unparseable. In such situations, the system will fall back to manual extraction using regular expressions.
  • Incomplete Data: The LLM may not always extract all the information from the document, especially if the document structure is unusual or the text quality is poor.

Database Errors

Database errors can occur during the saving of applicant or user data:

  • Connection Errors: Ensure that the database server is running and accessible.
  • Constraint Violations: If the data violates database constraints (e.g., unique constraints), the save operation will fail. Check the error message for details.
  • Transaction Errors: The application uses database transactions to ensure data consistency. If a transaction fails, all changes within that transaction will be rolled back.

Logging

The application uses Python's built-in logging module to record errors and important events. Logs are written to the console by default, but you can configure Django's logging settings to write logs to a file or a logging service.

To enable detailed logging, add the following to your settings.py:

LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'handlers': {
        'console': {
            'class': 'logging.StreamHandler',
        },
        'file': {
            'class': 'logging.FileHandler',
            'filename': 'ai_document.log',
        },
    },
    'loggers': {
        'ai_document': {
            'handlers': ['console', 'file'],
            'level': 'DEBUG',
        },
    },
}

Debugging Tips

  • Enable Debug Mode: Set DEBUG = True in your settings.py during development to see detailed error messages.
  • Check the Logs: Review the application logs for error messages and stack traces.
  • Test with Simple Documents: If processing fails, try testing with a simple, well-structured document to isolate the issue.
  • Verify Dependencies: Ensure that all required Python packages are installed and up to date.
  • Test the LLM: Verify that the Ollama service is running and that the specified model is available.

Security Considerations

When deploying this application in a production environment, it is essential to consider several security aspects to protect sensitive data and prevent unauthorized access.

File Upload Security

File uploads can be a significant security risk if not handled properly. The application implements several measures to mitigate these risks:

  • File Type Validation: Only PDF and DOCX files are allowed. The application validates the file extension and rejects other file types.
  • File Size Limits: A maximum file size of 50MB is enforced to prevent denial-of-service attacks through large file uploads.
  • Temporary File Storage: Uploaded files are stored temporarily and deleted after processing to minimize the risk of unauthorized access.
Additional Security Measures: Consider implementing virus scanning, content type verification, and secure file storage for production environments.

Data Protection

The application processes and stores sensitive personal information, including names, addresses, passport numbers, and health records. It is crucial to protect this data:

  • Encryption at Rest: Encrypt the database to protect data at rest. Most modern database systems support transparent data encryption.
  • Encryption in Transit: Use HTTPS to encrypt data transmitted between the client and the server. Ensure that your web server is configured with a valid SSL/TLS certificate.
  • Access Control: Implement robust authentication and authorization mechanisms to ensure that only authorized users can access the API endpoints. Django REST Framework provides several authentication classes that can be used for this purpose.
  • Data Minimization: Only collect and store the data that is necessary for the application's functionality. Avoid storing sensitive data that is not required.

Authentication and Authorization

The current implementation does not include authentication or authorization mechanisms. In a production environment, you should implement these features to protect the API endpoints:

  • Token-Based Authentication: Use token-based authentication (e.g., JWT) to authenticate API requests. Django REST Framework provides built-in support for token authentication.
  • Role-Based Access Control: Implement role-based access control to restrict access to certain endpoints based on the user's role. For example, only administrators should be able to access the batch conversion endpoint.
  • Rate Limiting: Implement rate limiting to prevent abuse of the API endpoints. Django REST Framework provides throttling classes that can be used for this purpose.

Input Validation

The application uses serializers to validate input data, but additional validation should be performed to prevent injection attacks:

  • SQL Injection: Django's ORM provides protection against SQL injection attacks by using parameterized queries. Avoid using raw SQL queries unless absolutely necessary.
  • Cross-Site Scripting (XSS): Sanitize user input to prevent XSS attacks. Django's template system automatically escapes output, but you should be careful when rendering user-generated content.
  • Command Injection: Avoid executing shell commands with user-provided input. If you must execute shell commands, use parameterized commands and validate the input carefully.

Logging and Monitoring

Implement comprehensive logging and monitoring to detect and respond to security incidents:

  • Audit Logs: Log all API requests, including the user, timestamp, endpoint, and response status. This information can be used to investigate security incidents.
  • Security Monitoring: Monitor the logs for suspicious activity, such as repeated failed login attempts or unusual API usage patterns.
  • Alerting: Set up alerts to notify administrators of potential security incidents in real-time.

Dependency Management

Keep all dependencies up to date to ensure that security vulnerabilities are patched:

  • Regular Updates: Regularly update Django, Django REST Framework, and all other dependencies to the latest versions.
  • Vulnerability Scanning: Use tools like pip-audit or safety to scan your dependencies for known security vulnerabilities.

Performance Optimization

For production deployments, several optimizations can improve the application's performance and scalability.

Asynchronous Processing

The document processing pipeline can be time-consuming, especially for large documents. To avoid blocking the main request thread and improve responsiveness, consider implementing asynchronous task processing using a task queue such as Celery. With Celery, the document upload endpoint can immediately return a response to the client, while the actual processing happens in the background. The client can then poll a status endpoint to check the processing status.

Caching

Implement caching to reduce database queries and improve response times for frequently accessed data. Django provides a flexible caching framework that supports multiple cache backends, including Redis and Memcached. Consider caching the results of expensive queries, such as the applicant list, and invalidating the cache when the data changes.

Database Optimization

Optimize database queries to reduce the load on the database server. Use Django's select_related and prefetch_related methods to reduce the number of database queries when accessing related objects. Create database indexes on frequently queried fields to speed up lookups. For the APIResponseLog model, indexes are already defined on key fields such as endpoint, user, and status_code.

File Storage

Future Enhancements

Currently, the application supports PDF and DOCX files. Future versions could add support for additional document formats, such as plain text files, images (using OCR), and HTML documents. This would make the application more versatile and capable of handling a wider range of input sources.

Improved Data Extraction

The current LLM-based extraction approach works well for structured documents but may struggle with highly unstructured or poorly formatted documents. Future enhancements could include:

  • Fine-tuned Models: Train or fine-tune LLMs specifically for seafarer application forms to improve extraction accuracy.
  • Multi-modal Models: Use multi-modal models that can process both text and images, allowing the system to extract information from scanned documents and forms with complex layouts.
  • Template Matching: Implement template matching algorithms to identify known document structures and extract data more reliably.

Real-time Processing Status

Implement a real-time status update mechanism using WebSockets or Server-Sent Events (SSE) to provide clients with live updates on the document processing status. This would improve the user experience by eliminating the need for polling.

Add support for document versioning, allowing users to upload multiple versions of the same document and track changes over time. This would be useful for maintaining a history of applicant information and tracking updates.

Advanced Search and Filtering

Enhance the applicant list endpoint with advanced search and filtering capabilities. Audit Trail

API Documentation

Generate interactive API documentation using tools like Swagger or ReDoc. This would make it easier for developers to understand and use the API endpoints. Django REST Framework provides built-in support for generating OpenAPI schemas, which can be used with these documentation tools.

Internationalization
